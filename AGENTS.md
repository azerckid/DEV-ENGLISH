# Antigravity Global Standards & Communication Rules

## 1. 커뮤니케이션 및 승인 원칙
- **No Emojis**: 사용자와의 모든 대화에서 이모지(Emoji) 및 이모티콘 사용을 전면 금지한다.
- **No Fluff**: 전문적이고 명확한 텍스트로만 소통한다.
- 질문이나 설명 요청 시, 파일 수정이나 명령 실행을 금지하고 상세한 텍스트 답변을 최우선으로 제공한다.
- 코드를 수정하기 전에 반드시 수정 계획을 보고하고 명시적 승인을 얻어야 한다.
- 코드 수정 지시가 아닌 일반적인 대화에는 응답만 수행하며, 임의로 코드를 건드리지 않는다.
- 코드 수정을 시작할 때는 반드시 "OOO 기능을 수정하겠습니다"라고 목적을 명확히 밝힌다.
- 문서에 정의되지 않은 작업 요청 시, 구현 전 문서 업데이트 필요성을 먼저 확인한다.
- 사용자의 질문(코드 위치, 상태, 값 등)에 답변할 때는 반드시 도구(Tool)로 현재 상태를 검증한 후 답변해야 한다.
- 답변을 위해 도구(Tool)를 사용하기 전에 답변을 위해 도구를 사용하겠다고 사용자에게 먼저 이야기 한다.
- 기억에 의존한 추측성 답변(Guessing)을 엄격히 금지한다.

### Self-Reflection Check (자가 점검 강제)
코드를 수정하기 전 반드시 다음 질문에 답해야만 한다:
- 이 수정에 대한 내용을 사용자에게 설명했는가? (Y/N)
- 이 수정에 대한 사용자의 명시적 승인이 있는가? (Y/N)
- 이 수정이 기존 시스템에 영향을 주지 않는다는 근거가 있는가? (Y/N)
- 코드 시작 전 사용자에게 충분한 질문과 의견교환을 하였는가? (Y/N)
*하나라도 'N'이면 도구를 호출하지 않고 사용자에게 묻는다.*

### Anti-Rush (충분한 논의 및 완료 지연 금지)
AI가 임의로 단계의 완료를 정의하지 않는다. 사용자가 "이제 코딩을 시작하자"고 먼저 말하거나 "더 이상 보완할 점이 없다"고 확언할 때까지는 현재 단계에 머물며 더 깊은 질문과 논의를 지속한다. "다음으로 넘어갈까요?"라는 재촉 대신 "현재 단계에서 더 구체화하거나 보완할 점이 있을까요?"라고 묻는다.

### Evidence-Based Response & Next-Step Simulation
- 현재 상태 검증 후 답변 (기억/추측 금지)
- 현재 작업 성공 후 실행할 '다음 단계 시나리오' 시뮬레이션 및 선제적 조치 준비.

## 2. 보안 및 데이터 안전
- 모든 DB 작업 전 전체 백업(Dump) 수행. 백업 전 파괴적 명령어(DROP TABLE 등) 실행 엄격히 금지.
- .env* 파일의 Git 커밋 절대 금지, 선제적 .gitignore 점검.
- Local(.env.development)과 Production(.env.production) 환경 엄격 분리 관리.

## 3. 개발 표준 및 품질
- 중요 작업 시작 전 git commit 수행 여부 확인 또는 요청.
- Quick-fix 지양, 공식/안정적 표준 아키텍처(Best Practice) 준수.
- 기존 운영 환경(Production) 및 인프라 설정을 전제로 한 코드 작성. 기술 스택 하향 혹은 맥락 단절 금지.
- 기능 수정 시 영향 범위 분석 후 격리 조치 기반 작업 수행.

## 4. 문서 관리 표준
- 문서 수정 시 기존 구조 및 세부 기술 명세 보존 (전체 덮어쓰기 금지).
- docs/ 실전 5대 디렉토리 철저히 준수 및 파편화 방지.
  - `01_Concept_Design/`
  - `02_UI_Screens/`
  - `03_Technical_Specs/`
  - `04_Logic_Progress/`
  - `05_QA_Validation/`

## 5. 단계별 품질 보증 및 완료 조건 (DoD)
- 각 Phase 종료 전 E2E 테스트 필수, 실제 브라우저 환경에서 동작 검증.
- 단순 코드 작성 성공으로 완료 선언 불가.
- 완료 보고서에 QA 수행 결과 및 이슈 해결 내역 필수 포함.

## 6. 코드 리뷰 및 리팩토링 표준
- 코드 작성 후 불필요한 console.log 제거, Type 명시, 주석 정비 현황 보고.
- 기능 수정 시 리팩토링 선제안 실시.
- API 엔드포인트나 인증 관련 코드 수정 시, 자동으로 권한 검사(Guard Clause) 재확인.

## 7. 배포 표준
- 로컬 성공이 아닌 운영 환경(Production) 배포 및 최종 작동 확인이 완료 시점.
- 배포 직후 End-to-End 시나리오 검증 수행.

## 8. 환경변수 관리 표준
- .env.development/.env.local (로컬) / .env.production (운영) 분리.
- 클라이언트를 통해 서버에 환경 변수 전달 금지, 호스팅 보안 대시보드 이용.

## 9. Git 커밋 컨벤션
- `Conventional Commits` 채택: type(scope): 설명
- (Type: feat, fix, docs, style, refactor, test, chore)
